## 1. 概要 / 目的

ユーザー自身の API キー（BYOK）を使って、エディタ上で AI 操作（要約・続き書き・翻訳・ページへの質問）を実行できるようにする。

このアプリの最大の差別化機能であり、キーはユーザーのブラウザ（localStorage）にのみ保存されアプリのサーバーには渡らないという設計の安心感を、UI/UX を通じて正直かつ丁寧に伝えることがデザイン上の最重要課題。

設計上の 2 つの軸:
- **エディタ内 AI メニュー**: 要約・続き書き・翻訳の 3 操作（結果プレビュー → 挿入/置換/破棄）
- **質問サイドパネル**: ページ全体を文脈にした Q&A（本文に自動挿入しない）

## 2. 画面構成

### 2-1. エディタページ全体（AI 操作中）デスクトップ

```
┌────────────────────────────────────────────────────────────────────────────────────────┐
│ [AppShell トップバー: h-10]  ≡  [保存済み ✓]             [AI 残回数: 本日 3/5 回]      │
├──────────────┬──────────────────────────────────────────┬──────────────────────────────┤
│              │         [エディタ領域]                    │  [AI 質問サイドパネル]        │
│  サイドバー  │  ┌──── max-w-[900px] mx-auto ────┐        │  w-80 border-l border-border  │
│  260px       │  │  [PageHeader: アイコン+タイトル] │        │  （質問操作のみ表示）          │
│  （M2 済み） │  │                                │        │  §2-4 を参照                  │
│              │  │  [BlockNote エディタ]           │        │                              │
│              │  │  ↑ テキスト選択中に            │        │                              │
│              │  │    フローティングツールバー      │        │                              │
│              │  │    が出現する                  │        │                              │
│              │  └────────────────────────────────┘        │                              │
└──────────────┴──────────────────────────────────────────┴──────────────────────────────┘
```

- AI 質問サイドパネルは「ページへの質問」操作を選択・送信したときのみ開く
- 閉じると `w-80` のスペースが消えてエディタが全幅に戻る
- デスクトップのみサイドパネルを右に表示。モバイルは下部ドロワーで表示（§2-5 参照）
- AI 残回数はトップバー右端に常時表示（エディタページのみ。設定ページでは非表示）

### 2-2. フローティングツールバーの AI 拡張（テキスト選択時）

M3 で実装済みのフローティングツールバー末尾に AI ボタンを追加する。

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│  B  │  I  │  U  │  S  │  </>  │  🔗  │ ▼ブロック変換  ┆  ✦ AI  ▾                    │
└──────────────────────────────────────────────────────────────────────────────────────┘
                                          ↑ M3 既存              ↑ M4 で追加
                                      セパレータ（Separator）で視覚分離
```

- 「✦ AI ▾」ボタン: `text-primary` の星形アイコン + "AI" ラベル + ドロップダウンアロー
- クリックでドロップダウンメニューが表示される（§3-1 参照）
- 鍵未登録時: ボタンを押すと「鍵未登録」トースト + 設定ページへのリンク（§6-3 参照）
- 鍵登録済みだが回数ゼロの時: ボタンを押すと「本日の上限に達しました」（§6-2 参照）

### 2-3. AI ドロップダウンメニュー

テキスト選択中に「✦ AI ▾」を押すと表示されるメニュー。

```
┌─────────────────────────────┐
│  ✦ AI                       │
├─────────────────────────────┤
│  要約する                   │
│  続きを書く                 │
│  翻訳する           ►       │
│  ─────────────────────────  │
│  このページに質問する       │
└─────────────────────────────┘
```

- 「翻訳する」はサブメニューを持つ（§3-2 参照）
- 「このページに質問する」は質問サイドパネルを開く
- 選択範囲の有無による操作変化:

| 操作 | 選択範囲あり | 選択範囲なし |
|------|------------|------------|
| 要約する | 選択テキストを要約 | ページ全体を要約 |
| 続きを書く | 選択テキストの末尾から続く | カーソル位置までの本文から続く |
| 翻訳する | 選択テキストを翻訳 | 選択が必要。メニューを押すと「翻訳するテキストを選択してください」と案内 |
| このページに質問する | 常にページ全体 + 質問文 | 同上 |

### 2-4. スラッシュメニューからの AI 起動

行頭の `/` からも AI 操作を呼び出せる。M3 の `SlashMenu.tsx` に AI グループを追加する。

```
─────── AI ────────────────────────────────────
  [✦]  要約する         ページまたは選択範囲を要約
  [✦]  続きを書く       現在位置から続きを生成
  [✦]  このページに質問  AI にページの内容を聞く
```

- スラッシュメニューの AI アイテムを選択すると「✦ AI」ドロップダウンと同じフローに入る
- 「翻訳する」はスラッシュメニューには含めない（テキスト選択が前提のため）

### 2-5. AI 質問サイドパネル

「このページに質問する」を選択すると表示される。

#### デスクトップ（md 以上）

```
┌─────────────────────────────────────────────────────────────┐
│  ✦ AI — このページに質問する              [×]               │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  [回答エリア: ScrollArea flex-1]                            │
│                                                             │
│  質問すると、ページの内容をもとに回答します。               │
│  （まだ質問がない場合のプレースホルダ）                     │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│  ┌─────────────────────────────────────────────────┐        │
│  │  質問を入力…                                    │        │
│  │                                                  │        │
│  └─────────────────────────────────────────────────┘        │
│  [送信]   ✦ AI が生成中は [停止]                            │
└─────────────────────────────────────────────────────────────┘
```

- 右パネルは `w-80`（320px）固定。`border-l border-border` で区切り
- `[×]` ボタンでパネルを閉じる。閉じると会話履歴はセッション中保持（ページリロードでリセット）
- 回答エリアは `ScrollArea` でスクロール可能
- 質問入力欄は `textarea`（`resize-none`、最大 2 行）。Enter で送信、Shift+Enter で改行

#### モバイル（md 未満）

サイドパネルではなく、画面下部から `Sheet`（ドロワー）で表示。
`side="bottom"`、高さ `h-[70vh]`。エディタの上に重なる形。

```
┌──────────────────────────────────────────────────────┐
│                   [エディタ]                          │
│                                                      │
├──────────── ▔▔▔▔▔ ドロップハンドル ▔▔▔▔▔ ────────────┤
│  ✦ AI — このページに質問する         [×]             │
│  ──────────────────────────────────────────────────  │
│  [回答エリア: ScrollArea flex-1]                     │
│  ──────────────────────────────────────────────────  │
│  [入力欄]  [送信]                                    │
└──────────────────────────────────────────────────────┘
```

- `Sheet` の `SheetContent side="bottom"` を流用（M2 実装の `components/ui/sheet.tsx`）
- ドロップハンドルをドラッグで高さ調整（shadcn/ui の Vaul 連携 / 実装は frontend 判断）
- モバイルではテキスト選択時のフローティングツールバーが表示されない場合があるため、PageHeader の右端に「✦ AI」ボタンを常時表示する（§2-6 参照）

### 2-6. モバイル専用 AI 起動ボタン

モバイルでは `PageHeader` エリアの右上に「✦ AI」ボタンを配置。

```
┌──────────────────────────────────────────────────────┐
│  [アイコン] 📄                       [✦ AI]          │
│  [タイトル入力欄]                                     │
│  ─────────────────────────────────────────────────── │
```

- PageHeader は M2 実装済みのため、変更箇所を frontend が判断すること

## 3. 4 操作の UX フロー

### 3-1. 要約（summarize）

**入力**: テキスト選択中 → 選択範囲。選択なし → ページ全体の `content_text`。

**フロー**:

```
① ユーザーが「要約する」を選択
        │
② [consumeAiUsage('gemini')] Server Action（回数チェック・消費）
        │ 失敗 → エラートースト（§6 参照）
        │ 成功
        ↓
③ [AiResultPreview] モーダルが開く
   ┌─────────────────────────────────────────────────────────┐
   │  ✦ 要約                                [×]             │
   │  ─────────────────────────────────────────────────────  │
   │  [ローディング中: Loader2 スピナー + "生成中..."]        │
   │    ↓ ストリーミング表示が始まる                         │
   │  [生成されたテキストがリアルタイムで表示される]          │
   │  ─────────────────────────────────────────────────────  │
   │             [破棄]    [ページ末尾に挿入]                │
   └─────────────────────────────────────────────────────────┘
        │
④ ユーザーが操作を選択
   ├─ 「ページ末尾に挿入」→ エディタの最終ブロックの後に挿入
   └─ 「破棄」 → モーダルを閉じる。エディタは変化なし
```

- ストリーミング表示: 各プロバイダの stream API を使い、生成中から逐次テキストを表示する（既存の `lib/ai/providers/*.ts` が対応する前提）
- 要約は「挿入のみ」（置換なし）。選択範囲の置換は翻訳に使用
- モーダル幅: `max-w-2xl`（デスクトップ）/ `max-w-[calc(100%-2rem)]`（モバイル）
- 生成中に `[×]` または `[破棄]` を押すとストリーミングを中断する（`AbortController` を使用。backend に要相談）

### 3-2. 続き書き（continue）

**入力**: カーソル位置までの本文（選択範囲がある場合も同じ。カーソル位置で判定）。

**フロー**:

```
① ユーザーが「続きを書く」を選択
② [consumeAiUsage] → ③ [AiResultPreview] モーダル
   ┌─────────────────────────────────────────────────────────┐
   │  ✦ 続きを書く                          [×]             │
   │  ─────────────────────────────────────────────────────  │
   │  [生成されたテキスト（ストリーミング）]                  │
   │  ─────────────────────────────────────────────────────  │
   │     [破棄]    [カーソル位置に挿入]                     │
   └─────────────────────────────────────────────────────────┘
④ 「カーソル位置に挿入」→ カーソルの直後にブロックとして追加
   「破棄」 → 変化なし
```

- 送る本文の最大文字数は `lib/constants/limits/` に定数で定義すること（backend に要相談）。超過時はトリミングするかエラーにするかを決めること

### 3-3. 翻訳（translate）

**入力**: 必ず選択範囲のみ（ページ全体翻訳は提供しない）。

**言語選択サブメニュー**:

```
┌──────────────────────────────────────────┐
│  翻訳する  ►    ┌────────────────────┐   │
│                 │  日本語 → 英語      │   │
│                 │  英語 → 日本語      │   │
│                 │  ─────────────────  │   │
│                 │  その他の言語を指定 │   │
│                 └────────────────────┘   │
└──────────────────────────────────────────┘
```

- 「その他の言語を指定」: インラインの小さな入力欄が展開（例: "フランス語"、"簡体中国語"）
- 最後に使った言語ペアを localStorage に記憶し、次回は先頭に表示する（`lib/ai/key-storage.ts` と同じファイルではなく別の `lib/ai/prefs-storage.ts` に実装。backend に要相談）

**フロー**:

```
① ユーザーが翻訳言語を選択
② [consumeAiUsage] → ③ [AiResultPreview] モーダル
   ┌─────────────────────────────────────────────────────────┐
   │  ✦ 翻訳（日本語 → 英語）               [×]             │
   │  ─────────────────────────────────────────────────────  │
   │  [元テキスト（選択範囲）の引用表示]                     │
   │  ─── ↓ ─────────────────────────────────────────────   │
   │  [翻訳されたテキスト（ストリーミング）]                  │
   │  ─────────────────────────────────────────────────────  │
   │     [破棄]    [選択範囲を置換]                         │
   └─────────────────────────────────────────────────────────┘
④ 「選択範囲を置換」→ 選択テキストを翻訳結果で上書き
   「破棄」 → 変化なし
```

- 元テキスト引用は `text-muted-foreground text-sm italic` でコンパクトに表示
- 翻訳は「置換のみ」（挿入オプションなし。翻訳した文を元文の隣に追加する UI は複雑すぎる）

### 3-4. ページへの質問（ask）

**入力**: ページ全体の `content_text` + ユーザーの質問文。

**フロー**:

```
① ユーザーが「このページに質問する」を選択
        │
② AI 質問サイドパネル（§2-5）が開く（既に開いていれば質問欄にフォーカス）
        │
③ ユーザーが質問を入力 → 「送信」ボタン or Enter
        │
④ [consumeAiUsage] Server Action
        │ 失敗 → 入力欄直下にエラーメッセージ（モーダルを開かない）
        │ 成功
        ↓
⑤ 回答エリアにアシスタントの回答がストリーミング表示される
   ┌─────────────────────────────────────────────────────────┐
   │  あなた  2026/06/14 14:32                              │
   │  この文書の要点を 3 点で教えてください                  │
   │                                                         │
   │  ✦ AI                                                   │
   │  1. ○○ ○○ ○○ ...（ストリーミング中）                   │
   └─────────────────────────────────────────────────────────┘
```

- 会話は**セッション中のみ保持**（ページリロードでリセット）。サーバーには送らない
- 本文への自動挿入なし。回答は読むためだけ
- 送信中は入力欄を `disabled`、「送信」を「停止」ボタンに変える
- 質問一覧は上から新しい順（最新が下。チャット UI として自然な順序）

## 4. 結果プレビュー UI（AiResultPreview）

### 4-1. モーダルの仕様

`components/ai/AiResultPreview.tsx`（Client Component）。既存の `components/ui/dialog.tsx` を流用。

```
┌───────────────────────────────────────────────────────────────┐
│  ✦ [操作名]                                     [×]           │
│  ─────────────────────────────────────────────────────────── │
│  [ScrollArea: max-h-[60vh]]                                   │
│  │                                                           │
│  │  [翻訳の場合のみ: 元テキスト引用]                         │
│  │  [生成テキスト: ストリーミングで順次追記]                  │
│  │                                                           │
│  ─────────────────────────────────────────────────────────── │
│  [破棄]         [挿入 / 置換]        ← 操作によってラベルが変わる │
└───────────────────────────────────────────────────────────────┘
```

### 4-2. ローディング状態

生成中（ストリーミング受信前・受信中）:

```
[Loader2 スピナー]  生成中...
```

- スピナー: `Loader2` (lucide-react) `animate-spin size-4 text-muted-foreground`
- ストリーミングが始まったらスピナーを消し、受信したテキストを随時表示

### 4-3. ストリーミング表示

- テキストは `whitespace-pre-wrap` でそのまま表示（段落・改行を保持）
- カーソル風のブリンクアニメーション（`|` を追記、最後の文字の後に表示）を任意で追加（実装は frontend 判断）
- 完了後はアニメーションを停止し、ボタンを活性化する

### 4-4. アクションボタン

| 操作 | 左ボタン | 右ボタン |
|------|---------|---------|
| 要約 | 破棄 | ページ末尾に挿入 |
| 続き書き | 破棄 | カーソル位置に挿入 |
| 翻訳 | 破棄 | 選択範囲を置換 |

- 生成中はどちらも `disabled`（ただし `[×]` ボタンは有効のまま。閉じると中断）
- 「破棄」: `variant="outline"`
- 挿入/置換: `variant="default"`（primary 色）
- ボタンはモーダル右下寄せ（`DialogFooter` の `sm:flex-row sm:justify-end`）

### 4-5. 差分表示

要約と続き書きは「挿入」なので差分表示は不要。
翻訳のみ「置換」のため、元テキストを引用表示することで差分を明示する。

翻訳モーダルの構成:

```
┌──────────────────────────────────────────────────────────────┐
│  元のテキスト                                                │
│  ────────────────────────────────────────────────────────   │
│  [元テキスト: text-muted-foreground italic text-sm]          │
│                                                              │
│  翻訳結果（日本語 → 英語）                                    │
│  ────────────────────────────────────────────────────────   │
│  [翻訳テキスト: text-foreground]                             │
└──────────────────────────────────────────────────────────────┘
```

## 6. エラー表示

### 6-1. 鍵が無効な場合

登録フローのテスト呼び出し失敗時:

- 場所: 鍵入力欄の直下
- スタイル: `text-destructive text-sm flex items-center gap-1`
- アイコン: `AlertCircle size-4`
- 文言: 「このキーは無効です。正しい API キーを確認してから再入力してください。」

エディタ内 AI 操作中に鍵が失効/削除された場合（プロバイダ API がエラーを返した場合）:

- 場所: トースト（画面下部）
- 文言: 「API キーが無効です。設定から再登録してください。」
- アクション: 「設定を開く」ボタン → `/settings/ai` に遷移

定数名案: `ERR_AI_KEY_INVALID`（`lib/constants/errors.ts` に追加）

### 6-2. 上限超過（回数ゼロ）

- 場所: トースト（画面下部）
- 文言:
  - 無料プラン: 「本日の AI 利用回数（5回）に達しました。明日 0 時（JST）にリセットされます。プレミアムプランでは 100 回/日ご利用いただけます。」
  - 有料プラン: 「本日の AI 利用回数（100回）に達しました。明日 0 時（JST）にリセットされます。」
- アクション（無料プランのみ）: 「プランを確認する」ボタン → `/settings` に遷移（第2弾で `/settings/subscription` に変更）

定数名案: `ERR_AI_DAILY_LIMIT_FREE = (limit: number) => ...`、`ERR_AI_DAILY_LIMIT_PAID = (limit: number) => ...`（`lib/constants/errors.ts` に追加）

### 6-3. 鍵未登録

AI メニューから操作しようとしたが、どのプロバイダの鍵も登録されていない場合:

- 場所: トースト（画面下部）
- 文言: 「AI を使うには API キーの登録が必要です。設定から Gemini などのキーを登録してください。」
- アクション: 「キーを登録する」ボタン → `/settings/ai` に遷移

定数名案: `ERR_AI_KEY_NOT_FOUND`（`lib/constants/errors.ts` に追加）

### 6-4. ネットワークエラー・プロバイダ API エラー

- 場所: `AiResultPreview` モーダル内（生成中に発生した場合）。または質問サイドパネルの入力欄直下
- 文言: 「通信エラーが発生しました。ネットワーク状態を確認して再試行してください。」
- アクション: モーダル内に「再試行」ボタン（再度 API 呼び出しを行う）

定数名案: `ERR_AI_NETWORK`（`lib/constants/errors.ts` に追加）

### 6-5. プロバイダ API のレート制限

プロバイダ側のレート制限（HTTP 429）:

- 文言: 「現在 API のリクエストが多すぎます。しばらく待ってから再試行してください。」

定数名案: `ERR_AI_RATE_LIMITED`（`lib/constants/errors.ts` に追加）

### 6-6. 無料プランで OpenAI/Anthropic を選択しようとした場合

UI では非表示にしているが、万一 `consumeAiUsage` でサーバー側に弾かれた場合:

- 文言: 「このプロバイダは有料プランでのみご利用いただけます。」（`ERR_PAID_REQUIRED` を流用）

## 7. BYOK の安心感を伝える文言

### 7-1. 設定ページのセキュリティバナー（§5-2）

正式文言:

> キーはこの端末のブラウザ（localStorage）にのみ保存されます。Notea のサーバーには送信・保存されません。
>
> PC とスマホなど、端末ごとに別々に登録が必要です。ブラウザのデータを消去するとキーも削除されます。

### 7-2. ヘルプページ（/help）での正直な説明

ヘルプの「API キーについて」セクション（UI ではなくコピー案として提供）:

**Q: API キーはどこに保存されますか？**

> Notea は API キーを Notea のサーバーや DB に保存しません。キーはあなたが今使っているブラウザの localStorage にのみ保存されます。これにより、仮に Notea のサーバーへの不正アクセスがあっても、あなたの API キーは守られます。
>
> ただし、キーのセキュリティはご利用の端末・ブラウザ環境の安全性に依存します。共用 PC では使用しないことをお勧めします。

**Q: AI を使ったとき Notea は何を記録しますか？**

> Notea のサーバーが記録するのは「今日何回 AI を利用したか」という回数だけです。あなたが入力した文章（プロンプト）や AI の回答内容は、Notea のサーバーには送られません。

**デザイン指針**: 誇張しない。「絶対安全」とは言わず、「サーバーに集めない構造」であることと「端末依存のリスク」の両方を正直に伝える。

## 9. データの流れ

### 9-1. AI 操作のデータフロー

```
[PageView]（Client Component）
  │ pageContentText（本文テキスト）を Editor から受け取る
  │
  ├─ [AiMenu]（Editor 内 Client）
  │   ├─ consumeAiUsage(provider)  → Server Action（鍵は渡さない）
  │   └─ lib/ai/providers/{provider}.generate(params)  → プロバイダ API 直接
  │       └─ 結果を AiResultPreview に渡す
  │
  └─ [AiAskPanel]（サイドパネル Client）
      ├─ consumeAiUsage(provider)  → Server Action
      └─ lib/ai/providers/{provider}.generate(params)  → プロバイダ API 直接
          └─ 結果をパネル内の回答エリアに表示
```

### 9-2. 設定ページのデータフロー

```
[app/(main)/settings/ai/page.tsx]（Server Component）
  └─ getProfile() → プラン取得
  └─ getAiUsageToday() → 今日の残回数
      └─ [AiKeyManager]（Client Component）
          └─ lib/ai/key-storage.ts（localStorage 読み書き）
      └─ [残回数パネル]（Client Component）
          └─ useAiUsage フックで定期更新
```

### 9-3. 残回数の更新タイミング

- エディタページのマウント時: `useAiUsage` が `getAiUsageToday` を呼ぶ
- AI 操作完了後（`consumeAiUsage` 成功後）: `useAiUsage` を手動で再取得
- `/settings/ai` 表示時: Server Component で取得（SSR）+ Client で `useAiUsage`
- 自動ポーリングはしない（不要。操作のたびに更新すれば十分）

## 11. エッジケース

### 11-1. 空ページ（本文なし）

- 要約: 「ページに本文がありません。テキストを入力してから要約してください。」→ トースト表示、API は呼ばない
- 続き書き: 「ページに本文がありません。テキストを入力してから続きを書かせてください。」→ トースト
- 翻訳: 選択範囲必須のため、そもそも発動しない（テキスト未選択時はボタンが disabled）
- 質問: ページ内容がないまま質問送信 → API には「（本文なし）」と渡す。ユーザーへの制限はしない

### 11-2. 本文が非常に長い場合

- プロバイダのコンテキスト上限を超えた場合はプロバイダ API がエラーを返す → `ERR_AI_NETWORK` として表示
- クライアント側でのトリミング上限は定数で管理する（backend/frontend に設計委ねる）

### 11-3. 生成中にページ遷移

- `AiResultPreview` モーダルが開いたままページ遷移すると、生成は中断される（`AbortController` を使用）
- 離脱確認ダイアログは表示しない（生成途中でも破棄ができるUIのため）

### 11-4. 複数プロバイダが登録されている場合

- どのプロバイダを使うかを自動選択する（優先順: 最後に使ったプロバイダ → 登録済みの中で最初のもの）
- 第1弾でのプロバイダ選択 UI は提供しない（複雑度を下げるため）
- ユーザーが登録している中から自動選択する処理を `lib/ai/index.ts` のディスパッチ層に実装すること（frontend に要相談）

### 11-5. ローディング状態（設定ページ）

- `AiKeyManager` マウント時: localStorage は同期読み取りのため即時表示。スケルトン不要
- 残回数パネル: Server Component で取得のため初期表示時は取得済み。再取得中は前の値を表示したまま

### 11-6. プロバイダ API の障害

- プロバイダ API がダウンしている場合: `ERR_AI_NETWORK` として表示し、ユーザーにプロバイダ側のステータスを確認するよう案内する
- 文言例: 「AI サービスへの接続に失敗しました。ネットワーク状態または各プロバイダのステータスページを確認してください。」

### 11-7. JST 0 時リセット前後

- リセット直前に回数がゼロになったユーザーが 0 時を跨いだ場合: 次の AI 操作時に `consumeAiUsage` が成功し、残回数表示が更新される（ポーリングなし）
