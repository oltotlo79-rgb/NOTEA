## 1. 概要 / 目的

M1〜M4 で構築した認証・ページ・エディタ・AI に、以下を追加して第1弾 MVP を完成させる。

| 追加機能 | URL |
|---------|-----|
| 設定ハブ（ナビ） | `/settings` |
| プロフィール設定 | `/settings/profile` |
| 使用量サマリー | `/settings/usage` |
| アカウント削除 | `/settings/account` |
| 外観設定 | `/settings/appearance` |
| ランディングページ本実装 | `/` |
| ヘルプ | `/help` |
| 利用規約 | `/terms` |
| プライバシーポリシー | `/privacy` |
| 特定商取引法に基づく表記 | `/tokushoho` |

既存の `/settings/ai`（M4 実装済み）と一貫したデザインを保つ。

**デスクトップ（md 以上）**

```
│               │  │ border-border     │                              │ │
```

**モバイル（md 未満）**

各設定ページではトップバーに「← 設定」リンクを表示して戻れる。

### 2-2. 設定ナビコンポーネント

`components/settings/SettingsNav.tsx`（Server Component）

現在の pathname と各設定ルートを比較してアクティブ項目をハイライトする。
ただし pathname の比較に `usePathname` を使う場合は Client Component になる。
`pathname` を親（Server Component の page.tsx）から props で渡すか、
専用の `SettingsNavClient.tsx` をリーフに切り出す設計を frontend が選択する。

```
ナビ項目の構成:
  プロフィール   → /settings/profile
  使用量         → /settings/usage
  外観           → /settings/appearance
  AI キー管理   → /settings/ai（M4 実装済み）
  アカウント     → /settings/account
```

各ナビ項目のスタイル:
- 通常: `flex items-center gap-2 px-3 py-2 rounded-md text-sm text-muted-foreground hover:bg-muted hover:text-foreground transition-colors`
- アクティブ: `bg-muted text-foreground font-medium`
- アイコン: lucide-react から各項目に合うアイコン（`User` / `BarChart2` / `Sun` / `Sparkles` / `Trash2`）`size-4`

### 2-3. 設定共通レイアウトコンポーネント

`components/settings/SettingsLayout.tsx`（Server Component）

`children` を受け取りナビ + コンテンツエリアを組み立てる。
`app/(main)/settings/layout.tsx` からこのコンポーネントを呼ぶ形にする。

props:
```
children: React.ReactNode
```

`/settings`（ハブ）自体はモバイル向けのタイル一覧を `SettingsLayout` とは別の形で実装してよい。
デスクトップでは `SettingsLayout` の右コンテンツ部分にハブコンテンツ（一覧）を表示する。

### 2-4. 設定ハブページ（/settings）

`app/(main)/settings/page.tsx`（Server Component）

デスクトップ: 設定ナビの右側に「プロフィール・使用量・外観・AI・アカウント」のカードグリッドを並べる。
モバイル: ナビ相当のタイル一覧。各タイルは設定サブページへのリンク。

```
  │  外観            │  │  AI キー管理     │
```

## 3. /settings/profile — プロフィール設定

### 3-1. 画面構成

```
┌──────────────────────────────────────────────────────────────────────┐
│  プロフィール                                                          │
│  ──────────────────────────────────────────────────────────────────  │
│                                                                      │
│  表示名                                                               │
│  ┌──────────────────────────────────────────────────────────────┐    │
│  │  田中 太郎                                                    │    │
│  └──────────────────────────────────────────────────────────────┘    │
│  50 文字まで                                                          │
│                                                                      │
│  [保存する]                                                           │
│                                                                      │
│  ──────────────────────────────────────────────────────────────────  │
│                                                                      │
│  メールアドレス                                                        │
│  ┌──────────────────────────────────────────────────────────────┐    │
│  │  user@example.com                              （変更不可）   │    │
│  └──────────────────────────────────────────────────────────────┘    │
│  メールアドレスの変更は現在サポートされていません。                   │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 3-2. コンポーネント分割

| コンポーネント | 配置 | SC/CC | 役割 |
|--------------|------|-------|------|
| `app/(main)/settings/profile/page.tsx` | ページ | **Server** | `getProfile()` で現在の表示名・メール取得。`ProfileForm` に初期値を渡す |
| `components/settings/ProfileForm.tsx` | フォーム | **Client** | 表示名入力・バリデーション・保存ボタン・成功/エラー表示 |

### 3-3. データの流れ

```
page.tsx（Server）
  └─ getProfile() → { displayName, email }
      └─ ProfileForm（Client）
          └─ updateProfile({ displayName })  ← Server Action
```

### 3-4. バリデーション仕様

| 項目 | ルール |
|------|-------|
| 表示名 | 1〜50文字、空白のみ不可 |
| バリデーションタイミング | 保存ボタン押下時（onBlur での途中検証は任意） |
| エラー表示位置 | 入力欄の直下 `text-destructive text-sm` |

### 3-5. 成功表示

保存成功後はトーストで「プロフィールを保存しました」を表示し、フォームは入力値を維持する（ページリロードしない）。
失敗時はトースト + 入力欄下にエラー文言。

### 3-6. アクセシビリティ

- `label` と `input` を関連付け（`htmlFor` + `id`）
- 入力欄: `aria-required="true"` / `aria-describedby` でヒントテキストを参照
- 変更不可メールアドレス: `<input readOnly aria-readonly="true" aria-label="メールアドレス（変更不可）">`
- 保存ボタン押下中: `aria-busy="true"` + ボタンを `disabled`

### 3-7. エッジケース

| 状態 | 挙動 |
|------|------|
| 初期表示名が未設定 | 入力欄は空。プレースホルダ「表示名を入力してください」 |
| 空白のみ入力して保存 | バリデーションエラー「表示名を入力してください」 |
| 50文字超過 | バリデーションエラー「50文字以内で入力してください」 |
| 通信エラー | トースト「保存に失敗しました。もう一度お試しください。」 |


### 4-2. コンポーネント分割

| コンポーネント | 配置 | SC/CC | 役割 |
|--------------|------|-------|------|
| `app/(main)/settings/usage/page.tsx` | ページ | **Server** | プラン・ページ数・ストレージ・AI回数を並行取得して各 Widget に渡す |
| `components/settings/UsageMeter.tsx` | 汎用 | **Server** 可 | プログレスバー + 数値表示の汎用メーター。label/used/limit/unit の props を取る |
| `components/settings/UsagePage.tsx` | ラッパー | **Server** 可 | 3つの `UsageMeter` と AI は `AiUsagePanel`（M4 実装済み）を並べるコンテナ |

`AiUsagePanel` は M4 で実装済み（`components/settings/AiUsagePanel.tsx`）。
AI 使用量の表示はこのコンポーネントを再利用し、`/settings/usage` にも同じ形で埋め込む。

### 4-3. データの流れ

```
page.tsx（Server）
  └─ Promise.all([
       getPageCount(userId),         // backend に要相談: count_user_pages rpc
       getStorageUsage(userId),       // 既存 lib/services/usage.ts
       getAiUsageToday(),             // 既存 lib/actions/ai.ts
       getProfile(userId)             // plan 取得
     ])
     └─ UsageMeter × 2 + AiUsagePanel
```

### 4-4. UsageMeter の表示仕様

#### プログレスバー

- 幅: `max-w-full`、高さ: `h-2`、角丸: `rounded-full`
- 背景: `bg-muted`
- 塗り: `bg-primary`。使用率 80% 以上で `bg-destructive`（警告色）
- 遷移: `transition-all duration-300`
- ARIA: `role="progressbar"` + `aria-valuenow` / `aria-valuemin` / `aria-valuemax` + `aria-label`

#### プラン別の上限表示

| プラン | ページ上限 | ストレージ上限 |
|--------|-----------|--------------|
| 無料 | 100 ページ | 200 MB |
| 有料 | 無制限 | 5 GB |

有料プランでページが「無制限」の場合: プログレスバーを表示しない。「プレミアムプランでは無制限です」と文言のみ表示。

#### ストレージ単位

1024 未満: MB 表示（小数点1桁）。1024 以上: GB 表示（小数点2桁）。
例: `84.3 MB`、`1.23 GB`

### 4-5. アップグレード案内

無料プランのみ使用量セクションの下部に表示:

```
上限を増やしたい場合はプレミアムプランをご検討ください。
（ページ無制限・ストレージ 5GB・AI 100回/日）
```

テキストリンクで `/settings` に遷移（第2弾で `/settings/subscription` に変更予定）。

### 4-6. エッジケース

| 状態 | 挙動 |
|------|------|
| ページ数 0 | バー幅 0%。「0 / 100 ページ」 |
| ストレージ 0 | 「0 MB / 200 MB」 |
| データ取得失敗 | 各メーターを `—` 表示にして「データを取得できませんでした」を表示 |
| 有料プランで上限なし | ページメーターは非表示。「無制限」テキストのみ |

## 5. /settings/account — アカウント設定

### 5-1. 画面構成

```
┌──────────────────────────────────────────────────────────────────────┐
│  アカウント                                                            │
│  ──────────────────────────────────────────────────────────────────  │
│                                                                      │
│  アカウントを削除                                                      │
│  ┌─────────────────────────────────────────────────────────────┐     │
│  │  アカウントを削除すると、すべてのページ・画像・データが      │     │
│  │  完全に削除されます。この操作は取り消せません。              │     │
│  │                                                              │     │
│  │  [アカウントを削除する]  ← variant="destructive" outline    │     │
│  └─────────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────────┘
```

### 5-2. 二段階確認ダイアログ

「アカウントを削除する」ボタンを押すと確認ダイアログが開く。

**ステップ 1（第1の確認）**

```
┌────────────────────────────────────────────────────────┐
│  アカウントを削除しますか？                              │
│  ──────────────────────────────────────────────────    │
│  この操作は取り消せません。以下がすべて削除されます:     │
│                                                        │
│  ・すべてのページとその内容                             │
│  ・アップロードした画像                                 │
│  ・登録した AI キー（ブラウザから）                     │
│  ・アカウント情報                                       │
│                                                        │
│  続けるには「アカウントを削除する」と入力してください。  │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │                                                  │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  [キャンセル]    [削除する]  ← 未入力時は disabled     │
└────────────────────────────────────────────────────────┘
```

- ダイアログ: `components/ui/dialog.tsx` を流用
- 確認テキスト入力欄: `components/ui/input.tsx`。`placeholder="アカウントを削除する"` とする
- 「削除する」ボタン: `variant="destructive"`。入力内容が「アカウントを削除する」と完全一致するまで `disabled`
- 入力一致チェックはクライアント側（`onChange`）で行い、一致した瞬間にボタンを活性化
- サーバー側 `deleteAccount` Action でも confirmation テキスト一致を再検証する（backend に要相談）

### 5-3. 削除実行中・完了後の遷移

| フェーズ | 表示 |
|---------|------|
| 削除ボタン押下後 | ボタン `disabled` + `aria-busy="true"`。「削除中…」テキストに切り替え |
| 削除成功 | ダイアログを閉じ、`/login` にリダイレクト |
| 削除失敗 | ダイアログ内にエラーメッセージ「削除に失敗しました。もう一度お試しください。」|

### 5-4. コンポーネント分割

| コンポーネント | 配置 | SC/CC | 役割 |
|--------------|------|-------|------|
| `app/(main)/settings/account/page.tsx` | ページ | **Server** | ページ構造のみ。データ取得不要 |
| `components/settings/DeleteAccountButton.tsx` | ボタン + ダイアログ | **Client** | 「削除」ボタン押下でダイアログ開閉・入力値管理・`deleteAccount` Action 呼び出し・リダイレクト |

### 5-5. アクセシビリティ

- ダイアログ: `role="alertdialog"` + `aria-labelledby` + `aria-describedby` を付与
- 入力欄: `aria-label="確認テキストを入力"` + `aria-required="true"`
- 削除ボタン: `aria-disabled="true"` を入力不一致時に付与
- ダイアログが開いたときフォーカスを入力欄に移動する
- `Escape` でダイアログを閉じる

## 6. /settings/appearance — 外観設定

### 6-1. 画面構成

```
┌──────────────────────────────────────────────────────────────────────┐
│  外観                                                                  │
│  ──────────────────────────────────────────────────────────────────  │
│                                                                      │
│  テーマ                                                               │
│  ┌─────────────────────────────────────────────────────────────┐     │
│  │                                                              │     │
│  │  [ライト]   [ダーク]   [システム]                            │     │
│  │   ○          ○          ●（選択中）                         │     │
│  │                                                              │     │
│  └─────────────────────────────────────────────────────────────┘     │
│  システムは OS の設定に合わせて自動的に切り替わります。               │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 6-2. テーマ選択 UI

3択ボタングループ。`role="radiogroup"` + 各ボタンに `role="radio"`。

| 選択肢 | 意味 |
|--------|------|
| ライト | 常に白背景 |
| ダーク | 常に黒/暗い背景 |
| システム | OS のダークモード設定に従う |

ボタンスタイル:
- 非選択: `border border-border rounded-lg p-4 flex flex-col items-center gap-2 hover:bg-muted cursor-pointer w-28`
- 選択中: 上記 + `ring-2 ring-primary`

各ボタン内部:
- アイコン: `Sun`（ライト） / `Moon`（ダーク） / `Monitor`（システム） `size-6`
- テキスト: ボタン名 `text-sm`

### 6-3. テーマの永続化

`next-themes` を使用（`ThemeProvider` を `app/layout.tsx` に追加。`attribute="class"` 戦略）。
テーマの保存先は `localStorage`（`next-themes` デフォルト）。
初回訪問時（未設定）: `システム` を初期選択として表示する。

### 6-4. コンポーネント分割

| コンポーネント | 配置 | SC/CC | 役割 |
|--------------|------|-------|------|
| `app/(main)/settings/appearance/page.tsx` | ページ | **Server** | `ThemeSelector` をレンダリングする薄い wrapper |
| `components/settings/ThemeSelector.tsx` | セレクタ | **Client** | `useTheme()`（next-themes）で現在テーマを取得・変更。`localStorage` の読み書きは next-themes が担う |

`ThemeSelector` は `useTheme` フックを使うため `'use client'` 必須。

### 6-5. アクセシビリティ

- `role="radiogroup"` + `aria-label="テーマを選択"`
- 各ボタン: `role="radio"` + `aria-checked={isSelected}` + `tabIndex`（選択中のみ 0、非選択は -1 で矢印キー移動）
- キーボード: `←` / `→` でボタン間移動、`Space` / `Enter` で選択

## 11. アクセシビリティ要点

### 11-1. スキップリンク

`app/(main)/layout.tsx` の AppShell 内、`<header>` の直前に配置する。
フォーカス時のみ表示（`sr-only focus:not-sr-only`）。

```
<a
  href="#main-content"
  className="sr-only focus:not-sr-only focus:fixed focus:top-4 focus:left-4
             focus:z-[100] focus:bg-background focus:border focus:border-border
             focus:rounded-md focus:px-4 focus:py-2 focus:text-sm focus:font-medium"
>
  メインコンテンツへスキップ
</a>
```

メインコンテンツの `<main>` 要素に `id="main-content"` を付与する。
公開ページ（`(public)` / `(legal)`）でも同様にスキップリンクを設ける。

### 11-2. フォーカス可視化

既存の shadcn/ui コンポーネントは `focus-visible:ring-3 focus-visible:ring-ring/50` を使用している。
カスタムコンポーネントでも同様のリングを付与すること。
フォーカスリングを `outline: none` で消す実装は禁止。

設定ナビのリンク: `focus-visible:ring-2 focus-visible:ring-ring` を明示的に追加する。

### 11-3. ARIA 要点

| 要素 | 必須の ARIA |
|------|------------|
| 設定ナビ | `<nav aria-label="設定メニュー">` |
| 設定ナビ各項目（アクティブ） | `aria-current="page"` |
| プログレスバー | `role="progressbar"` + `aria-valuenow` / `aria-valuemin` / `aria-valuemax` + `aria-label` |
| テーマ選択 | `role="radiogroup"` + `aria-label="テーマを選択"` |
| 各テーマボタン | `role="radio"` + `aria-checked` |
| 削除確認ダイアログ | `role="alertdialog"` + `aria-labelledby` + `aria-describedby` |
| 削除ボタン | `aria-disabled="true"` （未入力時） |
| 自動保存ステータス | `role="status"` + `aria-live="polite"`（既存 M3 実装済み） |
| LP ヒーロー見出し | `<h1>` でページ内 h1 は1つのみ |

### 11-4. キーボードショートカット

エディタページ（`/pages/[id]`）での新規ページ作成を Sidebar 経由でなく実行できるよう、
グローバルショートカットを設ける:

| 操作 | キー |
|------|------|
| 新規ページ作成 | `Ctrl/Cmd + N` |
| 設定を開く | `Ctrl/Cmd + ,` |
| サイドバー表示/非表示 | `Ctrl/Cmd + \` |

実装先: `hooks/use-global-shortcuts.ts`（Client Hook）。
`AppShell` 内で mount し、`document` に `keydown` リスナーを登録する。
入力欄・テキストエリアにフォーカスがある場合は発動しない。
`editor.isEditable` が `true` の場合は BlockNote の標準キーと衝突しないよう確認すること（frontend に要確認）。

ショートカット一覧は `/help` ページに掲載する。

| `text-muted-foreground` | WCAG AA 4.5:1 以上（実装者が実測すること） |
| エラーテキスト `text-destructive` | WCAG AA 4.5:1 以上 |
| プログレスバー塗り `bg-primary` | 視認性は background `bg-muted` との比率で確認 |
| LP ヒーローサブコピー | `text-muted-foreground` → 実装者が実測すること |

ダークモード時も同等の比率を保つことを `ThemeSelector` 実装時に確認する。

## 12. エッジケース（全体）

### 12-1. 設定ページ（ローディング / エラー）

各設定サブページは Server Component でデータ取得する。
データ取得中は `app/(main)/settings/loading.tsx` のスケルトンを表示する。
データ取得失敗は `app/(main)/settings/error.tsx`（Client Component、`'use client'` 必須）でキャッチし、`PageError` コンポーネント（M2 実装済み）を使う。

### 12-2. 権限なし

設定ページはすべて `(main)` グループのため、`app/(main)/layout.tsx` の認証チェックで守られる。
未認証アクセスは `/login` にリダイレクト（middleware 済み）。

### 12-3. LP（認証済みユーザーがアクセス）

第1弾では認証済みユーザーが `/` にアクセスしても LP をそのまま表示する（リダイレクトしない）。
CTA の「無料で始める」はログイン済みの場合 `/pages` に飛ばすかどうかは第2弾で検討。

### 12-4. 法務ページ（空白状態）

第1弾では法務ページはドラフトテキストで実装する。
「本内容は準備中です」等のプレースホルダは入れず、実際のドラフト文章を記載する（frontend エンジニアが雛形文言を実装すること）。
プライバシーポリシーの「収集しない情報」セクションは必ず実装する。

### 12-5. アカウント削除後のセッション

`deleteAccount` の実行後、Supabase Auth でユーザーが削除されるとセッションが無効化される。
削除成功後は `router.push(ROUTES.LOGIN)` で遷移し、`AppShell` が次のリクエストで未認証を検出してリダイレクトする。
Supabase セッションのクライアント側クリアを `deleteAccount` の呼び出し元（Client Component）で `supabase.auth.signOut()` を実行してから遷移することを推奨する（frontend に要相談）。
