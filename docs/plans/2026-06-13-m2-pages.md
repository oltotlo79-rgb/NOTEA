# M2: ページ管理 実装計画

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans。TDD（テスト→RED→実装→GREEN→commit）で進める。

**Goal:** ページの作成・取得・移動・並べ替え・ごみ箱（ソフトデリート/復元/完全削除）を Server Actions + RLS で実装し、サイドバーのページツリー UI と `/pages` `/pages/[id]` `/trash` を動く状態にする。

**Architecture:** 純粋なツリー操作ロジックを `lib/services/page-tree.ts` に隔離（単体テスト容易）。Server Actions は認証→Zod→制限チェック→DB の統一パターン。読み取りはRSCが `lib/queries/pages.ts`（`cache()`）、クライアントツリーは React Query（`['page-tree']`）が Server Action `getPageTree` を叩く。ごみ箱の cascade はスキーマ変更せず、最小カラム取得→サブツリー計算→単一 UPDATE で実現。

**Tech Stack:** Next.js 16 / React 19 / Supabase RLS / React Query 5 / Zod 4 / shadcn(Base UI)

**スコープ外（後続）:** BlockNote エディタ本体・自動保存・画像（M3）。ドラッグ&ドロップ並べ替え/付け替え UI（actions は実装・テスト済みにするが UI は後続。`movePage`/`reorderPage` は公開）。Sentry 連携（M5）。

## Task 2: page-tree サービス（純関数・TDD）

`lib/services/page-tree.ts`:
```ts
export type PageListItem = { id: string; parent_id: string | null; title: string; icon: string | null; sort_order: number }
export type PageTreeNode = PageListItem & { children: PageTreeNode[] }
type NodeRef = { id: string; parent_id: string | null }

export function buildPageTree(pages: PageListItem[]): PageTreeNode[]   // sort_order昇順→title でソート、親子に組み立て。孤児はルート扱い
export function getDescendantIds(pages: NodeRef[], rootId: string): string[]   // root除く子孫
export function getSubtreeIds(pages: NodeRef[], rootId: string): string[]      // root含む
export function getDepth(pages: NodeRef[], pageId: string): number             // ルート=1。循環時も無限ループしない(visited)
export function getSubtreeHeight(pages: NodeRef[], rootId: string): number     // 単一ノード=1
```
テスト: 平坦→木、複数ルート、ソート順、孤児、サブツリー、深さ、高さ、循環防御。

## Task 5: pages Server Actions（TDD）

`lib/actions/pages.ts`（`'use server'`、冒頭 @module に getPageTree/listTrashedPages の custom shape 理由を明記）。
- 共通: `requireUser` → Zod → 制限 → DB → revalidate。`.eq('user_id', userId)` を付与（多層防御）。
- `createPage({parentId?,title?})`→`ActionResult<{id}>`: 制限`create_page`。parentId 有時は全ページ(id,parent_id)取得→存在確認→`getDepth>=MAX_PAGE_DEPTH`拒否。末尾 sort_order=max+1。
- `updatePageContent({id,content,contentText})`: `JSON.stringify(content)` バイト>1MB 拒否。
- `updatePageMeta({id,title?,icon?})`。
- `movePage({id,newParentId})`: null→直接 `update parent_id=null`（循環/深さ不要）。非null→`rpc('move_page',{p_page_id,p_new_parent_id})`、message を ERR_PAGE_CIRCULAR/ERR_PAGE_DEPTH_LIMIT/ERR_DB に対応付け。
- `reorderPage({id,sortOrder})`。
- `trashPage(id)`: 非ごみ箱(id,parent_id)取得→存在確認→`getSubtreeIds`→`update is_trashed=true,trashed_at=now() .in(ids)`。revalidate pages+trash。
- `restorePage(id)`: ごみ箱(id,parent_id)取得→`getSubtreeIds`→無料は `count_user_pages + ids.length > FREE_MAX_PAGES` 拒否→親が消滅/ごみ箱なら root.parent_id=null→`update is_trashed=false,trashed_at=null .in(ids)`→必要なら root parent 修正。
- `deletePagePermanently(id)`: ごみ箱確認→`getSubtreeIds`→Storage `page-images` の各 `${userId}/${pageId}` を list&remove（M2は空=no-op）→`delete().eq('id',id)`（DBはcascade）。
- `getPageTree()`→`{data:PageListItem[];error?}`: 非ごみ箱 PAGE_LIST_SELECT sort_order昇順。
- `listTrashedPages(cursor?)`→`{data;nextCursor?;error?}`: is_trashed=true, trashed_at desc,id desc, limit TRASH_PAGE_SIZE, cursor は `lt('trashed_at',cursor)`。

テスト: 各 action の正常/未認証/Zod拒否/上限/移動エラー対応付け/サブツリー cascade を mock で網羅。test-utils を `_list/_single/_maybeSingle` の per-call override + storage モック対応に拡張（queryResult 名は後方互換維持）。
