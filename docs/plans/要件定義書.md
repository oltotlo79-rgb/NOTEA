# Notea 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト情報

| 項目 | 内容 |
|------|------|
| プロジェクト名 | Notea（ノーティア） |
| 概要 | Notion 風のメモ・ドキュメント作成 Web アプリ（BYOK 型 AI 機能付き） |
| 本番URL | 未定（Vercel ドメイン → 独自ドメイン検討） |
| リポジトリ | GitHub (プライベート) |
| ステータス | 企画・設計段階（2026-06-13 時点。実装未着手） |

### 1.2 目的

Notion のようなメモ・ドキュメント作成アプリをブラウザで提供し、以下を実現する：
- ブロック型エディタによる文書作成（見出し・リスト・コード等の装飾）
- ページの階層管理（ページの中にページを入れられるツリー構造）
- クラウド保存による PC・スマホ間の同期（ブラウザのみで両対応）
- AI 機能（要約・続き書き・翻訳・ページ内容への質問）の提供
- 月額課金（プレミアムプラン）による収益化（第2弾）

**このアプリの一番の特徴（差別化ポイント）**：
AI 機能はユーザー自身の API キー（BYOK: Bring Your Own Key）で動く。
- キーは**ユーザーのブラウザ（localStorage）にのみ保存**され、アプリのサーバー・DB は鍵を一切持たない
- AI の利用料はユーザー自身の負担となるため、**開発者側に AI のランニングコストが発生しない**
- サーバーが記録するのは「使った回数」だけ（鍵・プロンプト・応答は記録しない）

### 1.3 ターゲットユーザー

| ユーザー層 | 主な利用目的 |
|-----------|-------------|
| 無料の Notion 代替を探している人 | 日々のメモ・ドキュメント整理 |
| AI を自分のキーで使いたい人 | 自己負担で回数を気にせず AI 補助を使う（有料プラン） |
| PC・スマホ両方で使いたい人 | ブラウザだけで端末をまたいで同じメモを見る |
| Gemini 無料枠ユーザー | 無料キーで 1 日数回の AI 補助を使う（無料プラン） |

### 1.4 ビジネスモデル

| 収益源 | 状態 | 内容 |
|--------|------|------|
| プレミアム会員 | 第2弾で実装 | 月額300円 / 年額3,000円（Stripe） |
| 広告掲載 | 予定なし | — |

AI 利用料はどのプランでも「ユーザー自身の鍵＝本人負担」のため開発者に AI 費用はかからない。
「1日5回」等の制限は費用のためではなく、**有料プランへ誘導するための線引き**であり、数値は調整可能。

### 1.5 開発方針（段階リリース）

Notion の全機能を1人で最初から作るのは不可能。
**「一番大事な部分だけ作って世に出す → 反応を見て足していく」** 方式で進める。

| 段階 | 内容 | 目的 |
|------|------|------|
| 第1弾（MVP） | 基本のメモ機能＋ログイン＋基本AI（BYOK） | まず動くアプリを出す |
| 第2弾 | 表（データベース）機能・共有・検索・課金（Stripe） | 収益化と本格化 |
| 第3弾 | 複数人での同時編集・コメント・テンプレート等 | 機能の充実 |

### 2.5 インフラストラクチャ

ホスティングは **Vercel**。DB / 認証 / ストレージは Supabase に集約する。

| サービス | 用途 | プラン |
|---------|------|--------|

| Supabase | PostgreSQL / Auth / Storage | Free → 成長に応じて Pro |
| Stripe | 決済処理（第2弾） | 従量課金 |
| Sentry | エラー監視 | Free (5,000エラー/月) |
| GitHub Actions | CI（lint / test / build / e2e / CodeQL） | Free枠 |

#### デプロイ・スケジュール実行

- **デプロイ**: Vercel の GitHub 連携（`master` push で自動デプロイ、PR ごとにプレビュー環境）
- **Cron**: Vercel Cron（`vercel.json` で定義）から Bearer `CRON_SECRET` で叩く
  - `cleanup-trash`: 毎日 00:00 UTC（ごみ箱の 30 日経過ページを完全削除）
- **マイグレーション**: ローカルから `npx supabase db push`（リンク済みリモートに適用）。CI 自動適用は当面しない

### 2.6 サービス選定理由

| サービス | 選定理由 |
|---------|---------|
| Vercel | Next.js と相性が良く、無料から始められて公開が簡単。プレビュー環境が自動 |
| Supabase | DB（PostgreSQL + RLS）・ログイン（メール＋Google）・画像保管が**最初から1つで揃う**。1人開発の手間が最小 |
| BlockNote | Notion 風ブロックエディタをゼロから作るのは非現実的。既製部品で第1弾の最重要機能が一気に揃う。内部が Tiptap/ProseMirror ベースで、必要なら Tiptap への乗り換え余地もある |
| Stripe | 日本でも使える定番。月額・年額のサブスクリプションを作りやすい |
| Sentry | Next.js 公式対応、詳細なエラートラッキング |

#### Turso ではなく Supabase にした理由
- Turso は速くて安いが、ログイン・画像保存を**自分で別に組む必要があり**、1人の初心者には手間が多い
- 今回は「他の人にも使ってもらう」「ログインと画像が必須」のため、全部揃っている Supabase が向いている

## 3. 機能仕様

### 3.1 認証機能（第1弾）

#### 3.1.1 ユーザー登録
- メールアドレス + パスワードによる登録（Supabase Auth）
- メールアドレス確認: 登録後に確認メールを送信し、リンク踏破で有効化（Supabase Auth の Confirm email を有効化）
- パスワード要件:
  - 8文字以上
  - アルファベットと数字を各1文字以上含む（Zod でクライアント/サーバー双方検証）
- パスワードのハッシュ化は Supabase Auth に委任（bcrypt）
- 登録完了時に DB トリガー（`handle_new_user`）で `profiles` 行を自動作成（`plan = 'free'`）

#### 3.1.2 ログイン
- メールアドレス + パスワード認証（`signInWithPassword`）
- Google OAuth によるソーシャルログイン（`signInWithOAuth` → `/auth/callback` で `exchangeCodeForSession`）
- セッション管理: `@supabase/ssr` による cookie ベースセッション
- `proxy.ts` で毎リクエスト `updateSession()`（トークンリフレッシュ）
- 認可判定は必ず `supabase.auth.getUser()`（`getSession()` は検証なしのため使用禁止）

#### 3.1.3 パスワードリセット
- `resetPasswordForEmail` → メールのリンクから `/password-reset/confirm` で新パスワード設定
- レート制限は Supabase Auth 標準のもの（メール送信間隔制限）を利用

#### 3.1.4 ログアウト・アカウント削除
- ログアウト（全端末ログアウトは第2弾以降で検討）
- アカウント削除: 設定画面から自己削除可能（`pages` / Storage / `ai_usage` を cascade 削除。確認ダイアログ必須）

### 3.2 ページ管理機能（第1弾）

#### 3.2.1 ページの基本仕様

| 項目 | 仕様 |
|------|------|
| タイトル | 最大200文字（`MAX_PAGE_TITLE_LENGTH`）。空タイトルは「無題」と表示 |
| アイコン | 絵文字1文字（任意） |
| 本文 | BlockNote のブロック配列を `jsonb` で保存 |
| 本文サイズ上限 | 1ページあたり 1MB（`MAX_PAGE_CONTENT_BYTES`。超過時は保存エラー＋案内） |
| 階層 | `parent_id` による自己参照ツリー。最大深さ10（`MAX_PAGE_DEPTH`） |
| 並び順 | 同一親内で `sort_order`（手動並べ替え対応） |

#### 3.2.2 ページ操作
- 作成: サイドバーの「＋」/ ページ内の「サブページ追加」
- 移動: 親ページの変更（循環参照チェックを DB 関数 `move_page` で実施）
- 削除: **ソフトデリート（ごみ箱）**。`is_trashed = true` + `trashed_at` 記録
  - 子ページも連動してごみ箱へ
  - ごみ箱から復元可能（親がごみ箱内ならルートに復元）
  - ごみ箱で「完全に削除」を選ぶか、30日経過で cron が完全削除（`TRASH_RETENTION_DAYS = 30`）
- ページ数制限: 無料100ページ（ごみ箱内は数えない）。上限到達時は作成を拒否しアップグレード案内

#### 3.2.3 サイドバー（ページツリー）
- 全ページを1クエリで取得しクライアントで木に組み立て（React Query キー `['page-tree']`）
- 展開/折りたたみ状態は localStorage に保持
- 操作: 新規作成 / 名前変更 / 移動 / ごみ箱へ / サブページ追加（行のコンテキストメニュー）
- モバイルではドロワー表示

### 3.3 エディタ機能（第1弾）

#### 3.3.1 対応ブロック

| ブロック | 説明 |
|---------|------|
| 段落 | 通常テキスト |
| 見出し | H1 / H2 / H3 |
| 箇条書きリスト | ネスト対応 |
| 番号付きリスト | ネスト対応 |
| チェックリスト | チェック状態を保存 |
| 引用 | — |
| コードブロック | 言語指定＋シンタックスハイライト |
| 区切り線 | — |
| 画像 | アップロード（Supabase Storage）・貼り付け対応 |

- インライン装飾: 太字 / 斜体 / 下線 / 取り消し線 / インラインコード / リンク
- `/` コマンドメニューでブロック挿入（BlockNote 標準）
- BlockNote 標準にないブロック（引用・区切り線等）はカスタムブロックとして追加実装

#### 3.3.2 自動保存
- debounce 1.5秒（`AUTOSAVE_DEBOUNCE_MS = 1500`）で Server Action `updatePageContent` を呼ぶ
- 保存状態を常時表示: 「保存中… / 保存済み / 保存に失敗（再試行）」
- 保存失敗時はローカルの編集内容を保持したまま再試行（エラーバウンダリに投げない）
- 同一ページを複数タブで開いた場合は **last-write-wins**（第1弾では競合解決しない。第3弾のリアルタイム化で解消）

#### 3.3.3 画像アップロード

| 項目 | 仕様 |
|------|------|
| 対応形式（入力） | JPEG, PNG, WebP（GIF は非対応） |
| 受付上限（圧縮前の原画） | 20MB（`MAX_IMAGE_INPUT_SIZE_MB = 20`） |
| 圧縮 | **ブラウザ側で圧縮してからアップロード**: 長辺 2048px に縮小（`IMAGE_MAX_DIMENSION = 2048`）＋ WebP 品質 80 に変換（`IMAGE_WEBP_QUALITY = 80`）。実装は `lib/images/compress.ts`（ブラウザ専用、Canvas/OffscreenCanvas） |
| 保存形式 | WebP 統一（拡張子 `.webp`） |
| 保存上限（圧縮後） | 5MB（`MAX_IMAGE_STORED_SIZE_MB = 5`。圧縮失敗・異常ファイルへのガード。署名付きURL発行時にも検証） |
| 保存先 | Supabase Storage バケット `page-images`（非公開） |
| パス | `{user_id}/{page_id}/{uuid}.webp`（オリジナルファイル名は使わない） |
| セキュリティ | ファイルシグネチャ（マジックバイト）検証、MIME 一致確認 |
| 容量制限 | 無料 200MB / 有料 5GB（アップロード前に `getStorageUsage` でチェック。**カウントは圧縮後サイズ**） |
| 配信 | 署名付きURL（オーナーのみ閲覧可。RLS でユーザー別に隔離） |

- ブラウザ側圧縮を採用する理由: 署名付きURLでの直接アップロード設計と一致し、サーバー（Vercel）のリクエストボディ 4.5MB 制限を回避できる。Sharp 等のサーバー処理が不要になり、アップロード帯域も節約できる
- 効果の目安: スマホ写真 5MB → 約200〜500KB（無料 200MB で実質数百〜千枚程度）
- エディタ表示幅は最大約 900px のため、長辺 2048px は拡大表示にも十分な解像度

### 3.4 AI 機能（BYOK）（第1弾）

#### 3.4.1 提供する AI 操作

| 操作 | 入力 | 出力の扱い |
|------|------|----------|
| 要約 | ページ全体 or 選択範囲 | プレビュー表示 → 「挿入 / 置換 / 破棄」を選択 |
| 続き書き | カーソル位置までの本文 | 同上 |
| 翻訳 | 選択範囲（日→英 / 英→日 ほか） | 同上 |
| ページへの質問 | ページ全体＋質問文 | サイドパネルに回答表示（本文には自動挿入しない） |

#### 3.4.2 API キーの取り扱い（基本ルール）
- ユーザーは自分で取得した API キーを**自分のブラウザ（localStorage）に保存**する
- **アプリの DB にはキーを保存しない。アプリ側はキーを持たない**
- AI を使うときは、ブラウザから直接 AI プロバイダにキーを送って結果を受け取る形を基本にする（キーをサーバーに集めない）
- アプリ側で記録するのは**「使った回数」だけ**（キー・プロンプト・応答は記録しない）
- キーは cookie / URL / Server Action 引数に**絶対に載せない**
- 実装ルールの正本: `.claude/rules/ai-byok.md`

#### 3.4.3 利用フロー（必須順序）
1. ブラウザが Server Action `consumeAiUsage(provider)` を呼ぶ
   — 認証 → プラン判定（無料は Gemini のみ）→ 日次回数をアトミックに消費（上限超過なら拒否）
2. 成功したらブラウザが `lib/ai/providers/{provider}.ts` でプロバイダ API を直接呼ぶ
3. 結果はブラウザ内でエディタ／サイドパネルに反映。サーバーには送らない

回数チェックは必ず**呼び出し前**（後置きだと上限を超えられるため）。

#### 3.4.4 回数制限

| プラン | 1日の回数 | 登録できるキー |
|--------|----------|--------------|
| 無料 | 5回（`FREE_AI_DAILY_LIMIT`） | Gemini のみ |
| 有料 | 100回（`PAID_AI_DAILY_LIMIT`） | Gemini / OpenAI / Anthropic |

- 日付の境界は**日本時間（JST）の0時**にリセット（`ai_usage.used_on` は JST 基準の日付で記録。DB 関数内で `now() at time zone 'Asia/Tokyo'` から算出）。UI では「毎日0時にリセットされます」と案内
- プロバイダ制限は UI の出し分けに加えて**サーバー側（`consumeAiUsage`）でも検証**する

#### 3.4.5 キー管理 UI（設定 > AI）
- プロバイダ別にキーを登録・削除（入力欄は `type="password"`、表示は末尾4文字のみ）
- 登録時に形式チェック＋プロバイダへのテスト呼び出しで有効性検証
- 「キーはこの端末のブラウザにのみ保存されます。PC とスマホでは別々に登録が必要です」と明記
- キー取得手順へのリンク（各プロバイダの公式ページ）と簡単なガイドを表示

#### 3.4.6 正直な注意点（ヘルプに明記）
- ブラウザ保存のキーは「その端末／ブラウザの安全性」に依存する
- ただし「サーバーにキーを集めて一括で盗まれる」という最悪パターンは構造的に発生しない
- 個人開発でよく使われる、現実的でバランスの良い方式である

### 3.5 設定機能（第1弾）

|------|------|
| プロフィール | 表示名の変更 |

### 3.6 プレミアム会員機能（第2弾）

#### 3.6.1 料金プラン

| プラン | 価格（税込） | 備考 |
|--------|-------------|------|
| 月額プラン | 300円/月 | — |
| 年額プラン | 3,000円/年 | 2ヶ月分お得 |

#### 3.6.2 特典比較

| 項目 | 無料 | プレミアム |
|------|------|-----------|
| ページ数 | 100ページまで（`FREE_MAX_PAGES`） | 無制限 |
| 画像ストレージ | 200MB（`FREE_MAX_STORAGE_MB`） | 5GB（`PAID_MAX_STORAGE_GB`） |
| AI 回数 | 1日5回 | 1日100回 |

#### 3.6.3 決済
- Stripe Checkout による決済（クレジットカード）
- 自動更新（サブスクリプション）、カスタマーポータルで解約・カード変更
- Webhook（`/api/webhooks/stripe`）で `profiles.plan` を更新
- **Webhook 冪等性ガード**（`webhook_events` テーブル）で Stripe リトライによる重複処理を防止
- 解約時: 期間満了で `free` に戻す。上限超過分（101ページ目以降等）は**閲覧・編集は可、新規作成のみ不可**とする（データは消さない）

### 3.7 表（データベース）機能（第2弾）

- ページ内に表ブロックを挿入し、行・列（プロパティ）を定義
- プロパティ型: テキスト / 数値 / 選択肢 / 日付 / チェックボックス（初期セット）
- ビュー: テーブル / ボード（カンバン）/ カレンダー
- ※ 最も実装が重い機能。第2弾の中でも最後に着手し、仕様は実装前に別途詳細化する

### 3.8 共有機能（第2弾）

- ページ単位の共有リンク発行（閲覧のみ / 編集可）
- 共有相手: リンクを知っている人（第2弾）→ 特定ユーザー指定（第3弾で検討）
- RLS ポリシーを共有テーブル（`page_shares`）参照型に拡張

### 3.9 検索機能（第2弾）

- 検索対象: ページタイトル・本文テキスト（`content` から抽出したプレーンテキスト）
- 自分のページのみ検索（共有ページは第3弾で検討）
- PostgreSQL 全文検索（pg_trgm / pg_bigm）。`pages.content_text` カラムを保存時に同期

### 3.10 第3弾機能（概要のみ）

- 複数人での同時編集（リアルタイム。Supabase Realtime + CRDT 系ライブラリを検討）
- コメント・メンション
- テンプレート機能
- 詳細仕様は第2弾リリース後に策定する

### 3.11 SEO・公開ページ

- ランディングページ（`/`）に機能紹介・料金・BYOK の説明
- メタデータ: 全公開ページに title / description / OGP
- アプリ内ページ（`/pages/*`, `/settings` 等）は `robots: noindex`
- sitemap は公開ページのみ（アプリ内は含めない）

### 3.12 アクセシビリティ

- キーボード操作: エディタ・サイドバー・メニューを Tab / 矢印キーで操作可能に
- スキップリンク、ARIA labels、フォーカス表示強化
- コントラスト比 WCAG 2.1 AA 目標
- エディタ中心のアプリのため、キーボードショートカット（保存・新規ページ・検索）を用意

## 7. セキュリティ

### 7.1 認証・認可
- Supabase Auth（cookie ベースセッション、`@supabase/ssr`）
- `proxy.ts`（Edge）で毎リクエスト `updateSession()` ＋保護ルートのリダイレクト
- 認可判定は `supabase.auth.getUser()` のみ（`getSession()` 禁止）
- Server Action でも毎回 `requireUser()`（proxy に頼らない多層防御）

### 7.2 RLS（Row Level Security）
- **全テーブルで RLS 有効化必須**。基本は owner ポリシー（`auth.uid() = user_id`）
- Storage もユーザー別プレフィックスでポリシーを分離（公開バケットなし）
- `SUPABASE_SERVICE_ROLE_KEY`（RLS バイパス）は `lib/supabase/admin.ts` のみで使用し、webhook / cron / アカウント削除に限定
- anon キーは RLS 前提で公開可（`NEXT_PUBLIC_`）

### 7.3 AI キー（BYOK）の保護 — 本アプリ最重要
- キーの保存先は localStorage のみ。保存・読出しは `lib/ai/key-storage.ts` に集約（監査可能性の確保）
- キーを DB / サーバーログ / Sentry / cookie / URL / Server Action 引数に載せる経路を**設計として持たない**
- OpenAI パススルー（`/api/ai/proxy`）は唯一の例外的通過点:
  - Authorization ヘッダで受けて転送後破棄、`Cache-Control: no-store`
  - 転送先はプロバイダ公式ドメインの allowlist のみ（SSRF 防止）
  - 認証必須＋転送前に回数消費
- AI のプロンプト・応答はサーバーに保存しない（記録は回数のみ）
- レビュー・受け入れ評価では「キー漏洩経路」を重大度・高として必ず確認（evaluator エージェントの判定基準）

### 7.4 データ保護
- パスワードハッシュ・トークン管理は Supabase Auth に委任
- SQL インジェクション: supabase-js のパラメータ化クエリ（生 SQL を文字列結合しない）
- XSS: エディタ出力は BlockNote のスキーマ経由でのみレンダリング（任意 HTML を挿入させない）。`dangerouslySetInnerHTML` 禁止
- DB 関数は `security definer` 使用時に `search_path` を固定

### 7.5 ファイルアップロードセキュリティ
- ファイルシグネチャ（マジックバイト）検証、MIME と実体の一致確認
- UUID v4 ファイル名（オリジナル名不使用）
- 署名付き URL のパスをサーバー側で構築（クライアント指定パスを信用しない）
- 形式・サイズ・容量上限の三重チェック

### 7.6 入力検証
- 全 Server Action / Route Handler で Zod v4 による検証（境界で必ず検証してから使う）
- 制限値は `lib/constants/limits/` の定数に集約

### 7.7 レート制限・制限チェック
- AI 回数: `consume_ai_usage`（DB アトミック increment）
- ページ数・容量: 作成/アップロード前のチェック（`enforcePlanLimit`）
- 認証系（ログイン試行・メール送信）: Supabase Auth 標準のレート制限
- 第2弾以降、必要に応じて Upstash Redis による IP ベース制限を追加検討

### 7.8 セキュリティヘッダー（proxy.ts で付与）
- `Content-Security-Policy`: nonce 対応。`connect-src` に Supabase ＋ AI プロバイダ公式ドメイン
  （generativelanguage.googleapis.com / api.openai.com / api.anthropic.com）のみ許可
  — **CSP 自体が「キーが想定外のドメインへ送られない」ことの防衛線になる**
- `Strict-Transport-Security`（本番）
- `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`

### 7.9 Origin検証
- Server Actions の Origin ヘッダー検証（CSRF 対策。Next.js 標準＋proxy で補強）
- セッション cookie は Supabase 標準（httpOnly / SameSite=Lax / Secure）

### 7.10 監視・ログ
- Sentry によるエラー監視（AI 関連はエラー種別・プロバイダ名のみ。本文・キーは送らない）
- クライアントログは `lib/client-logger.ts` 経由（本番は console 抑制＋Sentry 送信）

## 8. パフォーマンス

### 8.1 最適化施策
- Server Components によるサーバーサイドレンダリング（エディタ枠・初期データ）
- BlockNote / AI プロバイダモジュールは `next/dynamic`（`ssr: false`）で遅延読み込み
- 自動保存は debounce 1.5 秒（入力ごとに保存しない）
- ページツリーは1クエリ取得＋クライアント構築（階層ごとの再帰クエリ禁止）
- カーソルベースページネーション（offset 不使用）
- React `cache()` / `unstable_cache`（ユーザー固有データはキーに userId 必須）
- `next/image` による画像最適化（Supabase Storage を remotePatterns 登録）

### 8.2 目標値

| 指標 | 目標 |
|------|------|
| ページエディタ初期表示（LCP） | 2.5秒以内 |
| ページ切り替え（ツリークリック→表示） | 1秒以内 |
| 自動保存の往復 | 500ms以内（体感は即時。ステータス表示で補完） |
| Lighthouse Performance（LP） | 90以上 |

| `e2e/trash.spec.ts` | ごみ箱（投入・復元・完全削除） |
| `e2e/ai-byok.spec.ts` | AI フロー（キー登録 → 回数消費 → プロバイダ stub 応答 → 挿入）。**ネットワーク傍受でキーが自アプリのサーバーに送られないことを検証** |
| `e2e/limits.spec.ts` | 無料プラン上限（100ページ・AI 5回）到達時の挙動 |
| `e2e/settings.spec.ts` | 設定（プロフィール・キー管理・使用量） |
| `e2e/public-pages.spec.ts` | LP・法務ページ・noindex |
| `e2e/responsive.spec.ts` | モバイルレイアウト（ドロワー） |
| `e2e/accessibility.spec.ts` | キーボード操作・ARIA |

### 9.4 AI 機能のテスト方針
- プロバイダ API は常にモック（`vi.stubGlobal('fetch')` / `page.route()`）。実呼び出し禁止
- 「キーがサーバーへ漏れない」ことを**テストで保証**:
  - `consumeAiUsage` の引数型・`ai_usage` 書き込み内容にキーが存在しない
  - `/api/ai/proxy` のレスポンス・エラーボディにキーが含まれない
  - E2E で自アプリドメインへのリクエストボディ/ヘッダにキー文字列が現れない

## 11. 環境変数

> 本番（Vercel）では Dashboard の Environment Variables で管理。`NEXT_PUBLIC_*` 以外はサーバー専用。

### 11.1 必須環境変数

```bash
# Supabase
NEXT_PUBLIC_SUPABASE_URL="https://xxxx.supabase.co"
NEXT_PUBLIC_SUPABASE_ANON_KEY="..."   # RLS 前提で公開可
SUPABASE_SERVICE_ROLE_KEY="..."       # RLS バイパス。lib/supabase/admin.ts 専用。絶対に NEXT_PUBLIC を付けない

# アプリケーション
NEXT_PUBLIC_APP_URL="https://..."

# エラー監視（Sentry）
SENTRY_DSN="https://..."
NEXT_PUBLIC_SENTRY_DSN="https://..."

# Cron（Vercel Cron から Bearer 認証で叩く）
CRON_SECRET="..."   # openssl rand -base64 32
```

### 11.2 オプション環境変数

```bash
# 決済（Stripe / 第2弾）
STRIPE_SECRET_KEY="..."
STRIPE_WEBHOOK_SECRET="..."
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY="..."
STRIPE_PRICE_ID_MONTHLY="..."
STRIPE_PRICE_ID_YEARLY="..."

# Google OAuth（Supabase Dashboard 側にも設定。ローカルは supabase/config.toml 経由）
GOOGLE_CLIENT_ID="..."
GOOGLE_CLIENT_SECRET="..."
```

### 11.3 存在しない環境変数（設計上の明記）

```bash
# AI プロバイダのキー用環境変数は存在しない。
# GEMINI_API_KEY / OPENAI_API_KEY / ANTHROPIC_API_KEY 等を追加した時点で BYOK 設計違反。
# キーはユーザーのブラウザ（localStorage）にのみ存在する。
```
